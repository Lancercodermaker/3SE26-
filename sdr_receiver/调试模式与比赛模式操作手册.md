# SDR Receiver 调试模式与比赛模式操作手册

日期: 2026-05-12

本文面向现场联调和比赛使用，说明 `sdr_receiver` 的两种运行模式、启动命令、键盘操作、仪表盘观察方法、profile 固化流程和常见故障处理。

## 1. 两种模式的边界

`sdr_receiver` 是单代码库、双运行模式:

- `run_mode=debug`: 人工调试模式。默认使用键盘控制，默认不由裁判上下文自动切换目标，默认不强制使用比赛 profile。目标是复现原 Python 接收端的手动调试逻辑，用于单独测试 INFO/L1/L2/L3、校准、调 gain、观察仪表盘。
- `run_mode=competition`: 比赛模式。禁用键盘控制，由 `/judge/radar_context` 或兼容 topic 自动驱动红蓝方和干扰等级，要求按 `match_slot/front_end_id/team/target` 加载 profile，解出 key 后发布 `/sdr/jam_code`，到最高破译等级后直接转入 INFO。

核心原则:

- Debug 模式服务于找参数和验证解调，允许人工切换。
- Competition 模式服务于上场闭环，不依赖人工键盘。
- Debug 模式下不要让比赛状态机偷偷覆盖手动目标，除非显式打开 `auto_context_control:=true`。
- Competition 模式下不要使用 `fallback_to_mock:=true`。

## 3. SDR 硬件检查

接收端 SDR 默认地址:

```text
ip:192.168.2.1
```

启动节点前先检查:

```bash
iio_info -u ip:192.168.2.1
```

如果看不到 `ad9361-phy` 或 `cf-ad9361-lpc`，先检查 USB 网卡、IP、电源、线缆和驱动，不要先怀疑 ROS2 节点。

## 4. Debug 模式启动

最常用的手动调试启动:

```bash
source /opt/ros/humble/setup.bash
cd ~/radar_ws
source install/setup.bash

ros2 run sdr_receiver sdr_receiver_node --ros-args \
  -p run_mode:=debug \
  -p auto_context_control:=false \
  -p use_real_sdr:=true \
  -p fallback_to_mock:=false \
  -p sdr_uri:=ip:192.168.2.1 \
  -p team:=RED \
  -p target:=L1 \
  -p terminal_ui:=true \
  -p publish_debug:=true \
  -p require_profile:=false
```

含义:

- `auto_context_control:=false`: debug 手动模式，不被裁判上下文强制切目标。
- `require_profile:=false`: profile 缺失时用内置默认参数。
- `target:=L1`: 初始监听 L1，可用键盘改。
- `terminal_ui:=true`: 打开终端仪表盘。

如果要在 debug 模式下测试比赛状态机，可以显式打开:

```bash
-p auto_context_control:=true
```

如果要在 debug 模式下测试 competition profile 是否能加载，可以显式打开:

```bash
-p use_profiles_in_debug:=true \
-p profile_path:=src/sdr_receiver/config/sdr_profiles/competition_profiles.yaml \
-p match_slot:=bo3_game1 \
-p front_end_id:=front_end_A
```

## 5. Debug 键盘操作

`sdr_receiver_node` 必须在前台交互终端运行，键盘才会生效。

```text
q       退出
r       切 RED
b       切 BLUE
1       INFO 普通模式
2       JAM L1
3       JAM L2
4       JAM L3
5       INFO-L3 rescue
6       INFO-L2 rescue
7       L2 direct preset
8       L3 direct preset
c       quick calibration；校准中再按则取消
f       full calibration；校准中再按则取消
[ / ]   切换当前 INFO rescue offset
- / +   当前 target gain 减/加 1 dB
```

推荐调试顺序:

1. 只开 L1 发射，按 `2`，看 `JAM L1` 是否显示 key。
2. 只开 L2 发射，按 `3`，看 `JAM L2` 是否显示 key。
3. 只开 L3 发射，按 `4`，看 `JAM L3` 是否显示 key。
4. 只开 INFO 发射，按 `1`，看 raw frame 和下半部数据是否更新。
5. 如果单独 INFO 解不出，按 `5` 测 INFO-L3 rescue，按 `6` 测 INFO-L2 rescue。
6. 如果 rescue 参数大致明确，用 `7` 或 `8` 直接套预设，再用 `[ / ]` 微调 offset。
7. 需要系统扫描时，再用 `c` 或 `f` 校准。

## 6. 仪表盘怎么看

仪表盘只初始化一次，然后原地刷新参数值。

上半部重点:

- `RED-L1 / BLUE-INFO`: 当前队伍和目标。
- `SEARCH / AC_ACTIVE / LOCKED`: 搜索、AC 命中、CRC16 锁定。
- `ADC / RMS`: 输入强度，过大可能饱和，过小可能听不到。
- `gain`: 当前增益。
- `AC / SOF / CRC8 / CRC16`: 解调链路逐级命中情况。
- `cmd`: 最近成功 CRC16 的命令，例如 `0xA06`、`0xA01`。
- `[CFG]`: 当前实际射频参数，包括 `lo`、`shift`、`gain`、`rf_bw`、`mode`。
- `Last frame`: 最近一帧来源、序号和 hex。
- `Error`: 最近错误或提示。

下半部重点:

- `HP / AMMO / COIN / POS / OCCU / BUFF`: INFO 解析结果。启动初始应全部为 0。
- `JAM L1/L2/L3`: 已解出的 key 和命中次数。比如 `JAM L1:[ABC123] 4` 表示 L1 key 已解出 4 次。

常见现象:

```text
AC 一直为 0:
  先检查 team、target、TX 模式、频点、SDR 连接。

AC 增长但 CRC16 为 0:
  多半是频偏、gain、rf_bw、filter、digital_shift 问题。

CRC16 有，但 JAM 行不显示 key:
  确认当前 target 是 L1/L2/L3，不是 INFO；也确认 payload 是可打印 6 字节 ASCII。

INFO raw_frame 有，但下半部数据不变:
  检查 cmd_id 是否为 0x0A01..0x0A05，以及 payload 长度和字段范围。
```

## 9. Competition 模式启动

比赛模式启动前必须确认:

- SDR 可访问。
- `/judge/radar_context` 有数据。
- `competition_profiles.yaml` 中当前 `match_slot/front_end_id/team/target` 全部存在。
- 雷达工程已经订阅 `/sdr/jam_code`，并负责把 key 发给裁判系统。

启动:

```bash
source /opt/ros/humble/setup.bash
cd ~/radar_ws
source install/setup.bash

ros2 run sdr_receiver sdr_receiver_node --ros-args \
  -p run_mode:=competition \
  -p use_real_sdr:=true \
  -p fallback_to_mock:=false \
  -p require_profile:=true \
  -p profile_path:=src/sdr_receiver/config/sdr_profiles/competition_profiles.yaml \
  -p match_slot:=bo3_game1 \
  -p front_end_id:=front_end_A \
  -p max_jam_break_level:=3 \
  -p terminal_ui:=true \
  -p publish_debug:=true
```

BO3 切换:

```bash
# 第二局
-p match_slot:=bo3_game2 -p front_end_id:=front_end_B

# 第三局
-p match_slot:=bo3_game3 -p front_end_id:=front_end_C
```

比赛模式下:

- 键盘控制会被禁用。
- `auto_context_control` 默认开启。
- `require_profile` 建议保持 true。
- `fallback_to_mock` 必须为 false。

## 10. Competition 状态机

输入来自裁判上下文:

- `self_id=9`: RED 雷达站。
- `self_id=109`: BLUE 雷达站。
- `radar_info_raw bit3-4`: 当前干扰等级，范围 `1..3`。
- `jam_level=0`: 比赛模式下视为无效，等待下一帧有效上下文。

期望流程:

```text
WaitingContext
  等待有效 self_id 和 jam_level

JamDecode L1/L2/L3
  根据当前 jam_level 选择 JAM_Lx_KEY profile
  解 0x0A06 key

WaitLevelUpdate
  如果当前 level < max_jam_break_level，发布 key 后等待裁判上下文等级提升

InfoDecode
  如果当前 level == max_jam_break_level，发布最终等级 key 后直接进入 INFO
```

例子:

```text
max_jam_break_level=3

裁判上下文 L1 -> 解 L1 key -> 发布 /sdr/jam_code level=1 -> 等 L2
裁判上下文 L2 -> 解 L2 key -> 发布 /sdr/jam_code level=2 -> 等 L3
裁判上下文 L3 -> 解 L3 key -> 发布 /sdr/jam_code level=3 -> 直接 INFO_UNDER_L3
```

如果 `max_jam_break_level=2`:

```text
L1 -> 解 L1 -> 等 L2
L2 -> 解 L2 -> 直接 INFO_UNDER_L2
```

## 11. Profile 维护

profile 文件:

```text
src/sdr_receiver/config/sdr_profiles/competition_profiles.yaml
```

层级:

```text
profile_sets:
  bo3_game1:
    front_end_A:
      RED:
        JAM_L1_KEY:
        JAM_L2_KEY:
        JAM_L3_KEY:
        INFO_UNDER_L1:
        INFO_UNDER_L2:
        INFO_UNDER_L3:
      BLUE:
        ...
```

每个 target 建议至少包含:

```yaml
rx_lo: 432580000
digital_shift: 80000
rf_bw: 660000
gain: 40
filter: hist248
```

从 debug 固化到 competition:

1. Debug 下人工找到 L1/L2/L3/INFO 的稳定参数。
2. 记录 `team`、`target`、`rx_lo`、`digital_shift`、`rf_bw`、`gain`、`filter`。
3. 写入当前 BO3 和前端编号对应位置。
4. 用 `run_mode=debug -p use_profiles_in_debug:=true` 复测 profile。
5. 再用 `run_mode=competition` 做 mock context 或真实裁判上下文测试。

## 12. 现场推荐流程

赛前调试:

1. 确认 `iio_info -u ip:192.168.2.1` 正常。
2. Debug 模式 `auto_context_control=false`。
3. 发射端单独发 L1，按 `2`，确认 `JAM L1` 有 key。
4. 单独发 L2，按 `3`，确认 `JAM L2` 有 key。
5. 单独发 L3，按 `4`，确认 `JAM L3` 有 key。
6. 单独发 INFO，按 `1`，确认 raw frame 和下半部数据更新。
7. 如果 INFO 不稳，按 `5/6/7/8` 试 rescue/preset。
8. 将稳定参数写入 profile。
9. 用 `use_profiles_in_debug=true` 复测 profile。
10. 用 mock 或真实 `/judge/radar_context` 跑 competition 流程。

上场前:

1. 确认 `match_slot`、`front_end_id`。
2. 确认当前 RED/BLUE 两边 profile 都有。
3. 确认雷达工程发布 `/judge/radar_context`。
4. 确认雷达工程订阅 `/sdr/jam_code`。
5. Competition 模式启动，`fallback_to_mock=false`。
6. 观察 `/sdr/status` 是否进入 `JamDecode`。

## 13. 常用排障表

```text
启动后一直 WaitingContext:
  competition 模式没有收到有效 /judge/radar_context。
  检查 self_id 是否为 9/109 或有效机器人 ID。
  检查 radar_info_raw bit3-4 是否为 1..3。

比赛模式提示 missing required SDR profile:
  当前 match_slot/front_end_id/team/target 缺 profile。
  不要临时关 require_profile 上场，先补 profile。

Debug 按 5/6/7/8 无效果:
  确认节点运行在前台交互终端。
  确认 run_mode=debug。
  确认 auto_context_control=false，避免裁判状态机覆盖手动 rescue。

JAM key 解出但雷达工程没反应:
  先 ros2 topic echo /sdr/jam_code。
  如果 topic 有数据，问题在雷达工程订阅或发裁判 key。
  如果 topic 没数据，回到解调链路检查。

L3 key 后进入 INFO_UNDER_L1:
  检查是否运行的是旧版本。
  新版本会用 submitted_key_level 或 rescue mode 保持 INFO_UNDER_L3。

INFO 单独无法解:
  先按 1 测普通 INFO。
  再按 5 测 INFO-L3 rescue。
  再按 6 测 INFO-L2 rescue。
  再按 7/8 套 direct preset。
  同时观察 AC/CRC8/CRC16 是卡在哪一级。

仪表盘重复打印整屏:
  检查是否运行的是旧版本。
  新版本使用备用屏原地刷新。
```
